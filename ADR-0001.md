## 背景与问题

当前 Hermes 有五层记忆：

| 层级 | 技术 | 数据量 | 感知管线角色 |
|------|------|--------|------------|
| L0 | MEMORY.md 索引 | 241 bytes，3条 | 全局索引兜底 |
| L1 | Palace SQLite FTS5（+ trigram） | 744 sessions，22,292 msgs | 关键词召回 |
| L2 | Obsidian wiki/Sources | 82 文件，24 compiled | wiki 检索 |
| L3 | gbrain SQLite 图 | 287 实体，1,079 关系 | 实体关系查询 |
| L4 | raw/articles | 剪藏原始资料 | 备用来源 |

**问题**：五层各自独立，agent 需要主动搜索才能调用。缺乏"在合适时机自动浮现相关上下文"的能力。

**目标**：在每次会话开头（session start），自动生成"可能相关的记忆摘要"，让 agent 主动想起而非被动搜索。

---

## 方案：会话感知摘要注入（Session-Aware Memory Summary Injection）

### 核心思路

在 session 开始时，由一个**感知管线**从五层检索相关内容，拼成一段结构化摘要，注入到 system prompt 的记忆区块。

```
Session Start（触发条件见下方）
    ↓
感知管线（Perception Pipeline）
    ↓
L0 MEMORY.md → 取出索引关键词（始终读取，全局兜底）
L1 Palace FTS5（+trigram） → 基于用户消息做 top-K sessions 召回
L2 wiki/Sources → 基于关键词检索相关 wiki 页面
L3 gbrain → 查实体关系图谱，输出关联实体列表
    ↓
拼装摘要（结构化 markdown，max 1500 字符）
    ↓
注入 system prompt 的记忆区块
    ↓
Agent 在对话中自然使用这些上下文
```

> **召回精度说明**：L1 的召回是 FTS5+trigram 关键词/子串匹配，不是真正的语义向量检索。召回精度有限，摘要结果应标注"以下来自关键词匹配，可能不完整"。

### Session 触发条件（精确定义）

触发感知管线的条件：**每次新的用户会话发起时**，具体包括：

- 用户发起新对话（`/new` 或 `/reset`）
- 飞书用户发来第一条消息（该用户之前无活跃 session 或上一 session 已结束）
- cron 任务以"primary"身份首次响应用户

**不触发**：同一 session 内的后续消息（保持低延迟）、后台子 agent 执行。

### 感知管线输入契约

```python
{
    "user_message": str,          # session 首条用户消息
    "session_id": str,            # 当前 session id（用于避免重复注入自身）
    "workspace_hint": str = "",   # 工作区类型枚举，见下方
    "max_chars": int = 1500,     # 输出上限字符数
    "timeout_ms": int = 1500      # 总 wall-clock 超时
}
```

**workspace_hint 枚举值**（v6 扩展）：
> **注意**：MVP 阶段（L0+L1）暂不使用 workspace_hint 参数，全量检索。后续扩展 L2/L3 时启用。

- `""`（空）— 无特定工作区，使用默认检索策略
- `"obsidian-vault"` — 当前工作区为 Obsidian vault
- `"git-repo"` — 当前工作区为 git 仓库
- `"hermes-home"` — 当前工作区为 Hermes 配置目录
- `"feishu"` — 当前通过飞书交互
- `"cli"` — 交互式命令行

> 若为空字符串，感知管线使用全量检索不过滤工作区上下文。

### 感知管线输出格式

> **v1 格式 vs ADR 目标格式说明**：ADR v6 锁定的输出格式为下方「ADR 定义格式」，当前脚本 `perceive-memory.py` 输出的 v1 格式（`### Facts（L0 记忆索引）` / `### Facts（L1 最近相关会话）`）为 MVP 阶段的过渡格式，两者的对应关系为：
> - ADR `### Facts` → v1 `### Facts（L0 …）` / `### Facts（L1 …）`
> - ADR `[来源: Lx] [high/medium/low]` → **v1 暂不实现**，MVP 验收时不检查
> - ADR `### Related Entities` → v1 `### Related Entities`（保留 session 来源信息）
> - ADR 目标格式为 v2 扩展方向，不阻塞当前 MVP 交付

```markdown
## 感知记忆摘要

> 以下结果来自关键词匹配（L0-L4），可能不完整，仅作参考。

### Facts（来自记忆检索，请引用时标注来源）
- [来源: L0/L1/L2/L3/L4] [high/medium/low] 内容

### Related Entities（图谱关联）
- 实体名（类型）：关联说明

<!-- truncated -->

> **可靠性标注**：MVP 阶段暂不实现。可靠性 `[high/medium/low]` 标注待召回质量评估完成后（中期考虑）再加入。
```

**可靠性标注**（hermes-win 补充问题 #4）：
- `[high]` — L3 gbrain 关系数据，经实体消歧
- `[medium]` — L1/L2 关键词匹配，上下文完整
- `[low]` — L0 索引召回，可能不完整

> 模型不得将感知摘要中的 `[low]` 信息当作事实输出。

### 注入机制（v5 锁定）

**方案：文件注入（方案 A）**

| 项目 | 说明 |
|------|------|
| 调用方 | `run_agent.py` |
| 调用方式 | subprocess：`python3 /path/to/hermes-memory-awareness/scripts/perceive-memory.py <json_input>` |
| 幂等性 | `run_agent.py` 先检查 `~/.hermes/cache/perception/{session_id}.injected` 标记文件，已注入则跳过 |
| 降级 | 若感知管线全部超时，写入降级摘要：`[记忆检索超时，仅 L0 索引供参考]` |

**注入流程：**

```
session 启动
    ↓
run_agent.py 检查 {session_id}.injected 标记
    ├── 已存在 → 跳过感知管线，直接启动 agent
    └── 不存在
            ↓
        subprocess 调用 perceive-memory.py
            ↓
        perceive-memory.py 写入 {session_id}.summary
            ↓
        run_agent.py 读取摘要，拼入 system prompt
            ↓
        写入 {session_id}.injected 标记
            ↓
        启动 agent
```

**perceive-memory.py 输出文件格式：**

```
~/.hermes/cache/perception/
  {session_id}.summary   # 感知摘要内容（markdown）
  {session_id}.injected   # 幂等标记（空文件，mtime=注入时间）
```

**subprocess 调用开销处理：** Python 冷启动约 200ms，计入 L0 读取的 100ms 预算内，总感知管线超时仍为 1500ms。

### 注入位置与优先级

感知摘要注入到 system prompt 的 **记忆区块（Memory Context）**，位于：
1. SOUL.md 身份定义之后
2. 用户 MEMORY.md 索引之前

**优先级规则**：
- 用户即时指令 > 感知摘要
- 感知摘要与即时指令冲突时，以即时指令为准（感知摘要仅作背景参考）
- 模型不得将感知摘要中的未验证信息当作事实输出

|---------|------|

### 数据一致性声明（hermes-win 补充问题 #1）

> **重要**：感知管线在 cron 任务运行间隙执行时，可能读到跨层不一致的数据（如 L1 有新 session 但 L2 还没同步编译）。

感知摘要中标注各层数据的**最后同步时间**（格式：`last_sync: YYYY-MM-DD HH:mm`）。

若某层数据超过 `stale_threshold` 未更新，标注 `[可能过时]`。

**分层阈值**：
- L1（Palace）：24h（每日 sync）
- L2（wiki/Sources）：7d（每周一评分）
- L3（gbrain）：7d（每周一抽取）
- L4（MEMORY.md）/ L5（raw/articles）：不设 stale 检查

感知管线**不保证跨层数据一致性**，结果仅供参考。

### L0 索引生命周期管理（hermes-win 补充问题 #2）

L0 MEMORY.md 索引当前 241 bytes / 3 条，但随时间膨胀。

**L0 淘汰策略**：
- L0 索引保留**最近 N 条**（N=10）+ **高频引用条**（引用计数 ≥3）
- 超出部分下沉到 L1，作为普通 session 记录检索
- L0 索引总大小上限 2000 字符，超出时触发淘汰

> 若 MEMORY.md 接近 2200 字符（memory 工具上限），感知管线自动跳过 L0 读取避免触发截断。

### 敏感信息过滤（必须实现）

**结构性风险**：Palace（744 sessions）、wiki 可能有密钥、内部 URL、个人数据。注入 system prompt 前必须完成过滤。

过滤规则详见：`references/sensitive-patterns.md`

核心规则（ADR 内联摘要）：
1. **密钥前缀**（精确）：`sk-`、`ghp_`、`AKIA`、`api_key=`、`password=`、`secret=`、`token=`、`Authorization: Bearer`
2. **格式正则**：base64 key、飞书群 ID（`oc_[a-z0-9]{20,}`）、内部 IP（`10.x.x.x`、`192.168.x.x`）
3. **私钥标记**：`-----BEGIN PRIVATE KEY-----`、`-----BEGIN RSA PRIVATE KEY-----`
4. **数据库连接串**：`postgres://`、`mysql://`、`mongodb://`、`redis://`
5. **字段名规则**（MVP 除外）：`api_key=`、`password=`、`secret=`、`token=` 等字段值，ADR v6 决策：MVP 阶段暂不实现字段名匹配，由敏感信息治理层面统一处理

> 完整规则在 `references/sensitive-patterns.md`，ADR 只引用不内联。

### 缓存机制

**目的**：避免相同/相似 user_message 重复跑五层检索。

```
Cache key: SHA256(normalized_user_message)[:16]
TTL: 1 小时
Max entries: 128（LRU 淘汰）
存储位置: ~/.hermes/cache/perception/
```

**降级**：缓存命中时直接返回，不走 L1/L2/L3/L4 检索。

### 感知管线性能预算（hermes-win 补充问题 #7）

| 阶段 | 时间预算 |
|------|---------|
| L0 读取 | 100ms |
| L1 FTS5 搜索 | 500ms |
| L2 wiki 搜索 | 400ms |
| L3 gbrain 查询 | 300ms |
| 拼装 + 过滤 + 截断 | 200ms |
| **合计** | **1500ms** |

### 失败与降级策略

| 故障情况 | 处理方式 |
|---------|---------|
| L1/L2/L3/L4 单源超时（> timeout_ms） | 静默跳过该源，其他源正常返回 |
| 全部超时（总耗时 > timeout_ms） | 返回降级摘要：`[记忆检索超时，仅 L0 索引供参考]` |
| L0 读取失败 | 返回空摘要，不阻断 session 启动 |
| 缓存命中 | 直接返回缓存结果，跳过检索 |
| 任何源报错 | 该源记入 error log，不阻断其他源 |
| 跨层数据不一致 | 在摘要中标注 `last_sync` 时间，不阻断返回 |

### 观测与审计

每次感知管线执行后，记录 JSON Lines 到 `~/.hermes/logs/perception.log`：

```json
{
    "timestamp": "2026-05-17T10:00:00",
    "session_id": "xxx",
    "cache_hit": false,
    "wall_time_ms": 1200,
    "sources_used": ["L0", "L1", "L2", "L3"],
    "items_returned": {"L0": 3, "L1": 5, "L2": 2, "L3": 4},
    "chars_output": 890,
    "redacted_count": 0,
    "truncated": false,
    "stale_sources": [],
    "error": null
}
```

> 日志不记录实际召回内容（防止敏感信息二次泄漏），只记录统计信息与来源标签。

## 决策

1. **项目位置**：在现有 Obsidian 仓库的 `wiki/projects/hermes-memory-awareness/` 下开展，不新建 vault
2. **感知管线**：纯本地脚本 `perceive-memory.py`，MVP 优先实现，L0+L1 先跑通再扩展
3. **Session 触发**：新会话发起时触发（包括飞书第一条消息），同一 session 后续消息不触发
4. **摘要长度**：上限 1500 字符
5. **路径约定**：使用 `~/.hermes/`、`~/obsidian-vault/` 等环境变量展开，不硬编码
6. **敏感信息过滤**：必须在感知管线内联完成，规则在 `references/sensitive-patterns.md` 集中维护
7. **缓存**：LRU，TTL 1小时，max 128 条，缓存 key 为 message hash
8. **失败降级**：单源超时静默跳过，全部超时返回降级提示，不阻断 session
9. **观测审计**：每次执行记 JSON Lines 日志，仅统计信息不记录召回内容
10. **接口锁定**：通过 ADR 文件锁定接口规范，脚本版本与 ADR 版本绑定（ADR-0001 v5 → perceive-memory.py v1）
11. **L0 生命周期**：总上限 2000 字符，超出时淘汰最久未访问条目（LRU），读取时动态过滤不写入磁盘
12. **数据一致性**：摘要标注各层 last_sync 时间；stale_threshold：L1=24h，L2=7d，L3=7d
13. **workspace_hint**：支持枚举值 `"" / "obsidian-vault" / "git-repo" / "hermes-home"`
14. **MVP 成功标准**：
    - L0 读取：MEMORY.md 内容全部成功读取（不超过 2200 字符）
    - L1 召回：给定 10 条测试 query，top-5 召回结果与手动搜索对比 Recall@5 ≥ 60%
    - 端到端：感知摘要注入后，model 响应中自然使用摘要上下文（LLM-as-judge 主观评估 ≥ 3/5）
    - 敏感过滤：744 sessions 中随机抽样 5 条，包含密钥的 session 不出现在召回结果中
15. **注入机制**：方案 A（文件注入），见「注入机制（v5 锁定）」节

### 注入函数

**`_inject_perception_summary(agent, user_message)`**

- 检查 `{session_id}.injected` 标记 → 幂等跳过，直接读 summary 重注
- 脚本不存在或调用失败/超时（3s）→ 静默降级，不阻塞会话
- subprocess 返回 0 → 读 `{session_id}.summary`，调用 `_inject_into_system_prompt` 注入

**`_inject_into_system_prompt(agent, summary_text)`**

- **注入位置**：SOUL.md 身份定义之后、`Conversation started:` 时间戳之前
- **幂等性**：检查 `summary_text.strip() in current`，已存在则跳过
- **兜底**：找不到 marker 时追加到 prompt 末尾

### 实现决策记录

| 决策 | 方案 | 原因 |
|------|------|------|
| 注入在 `_restore_or_build_system_prompt` 之后 | `_inject_perception_summary` | system prompt 已构建完毕，可安全注入 |
| subprocess timeout=3s | 硬编码 | Python 冷启动约 200ms，3s 足够覆盖感知管线 1500ms 预算 |
| `env={}` 传空环境变量 | subprocess.run | 避免敏感 env 变量泄露到感知管线子进程 |
| `sys.executable` 而非 `python3` | subprocess argv[0] | 跨平台兼容，Windows 下 `python3` 可能解析为 Store stub |
